openapi-lark v0.3 — AI 可读 / Recall 设计决策记录

日期 2026-05-22  ·  状态 SCRAPPED 设计被砍  ·  替代方案 DONE openapi-lark init 自动注入 CLAUDE.md 段

1. 文档作用

本文档是一份决策记录，不是实现 spec。原始设计稿（manifest.json + recall CLI 子命令 + lark-cli --live 兜底）经过 5 轮 codex 评审后，因前提未验证而被砍。保留本文以便将来真有大 spec / 跨 service 场景时，作为设计参考起点。

2. 原始问题陈述

「AI 在 Claude Code 里写代码时不方便查项目自己的 API spec — 飞书 wiki 的人可读 markdown AI 解析丢字段；飞书 wiki 自带搜索精度差；schema $ref 链 AI 追起来烦」

3. 原始设计简述（被砍）

  1. 每次 sync 末尾产 .openapi-lark/manifest.json：含 dereferenced schema、endpoint 元数据、本地 md 路径、wiki URL、shape 字段索引
  2. openapi-lark 长出 recall 子命令组：list-services / list-endpoints / search / get
  3. get --live 模式走 lark-cli docs +fetch --api-version v2 拉飞书最新
  4. Skill 触发词 + JSON 输出让 AI 通过 Bash 工具自助查询

4. 砍掉的理由（实测推翻前提）

4.1 验证 1：飞书 search 精度并不差

原设计稿假设 lark-cli drive +search 不能用于技术 spec。实测 2026-05-22 推翻这个假设：

$ lark-cli drive +search --query "predicts"
# total: 12 — 命中 POST /api/v1/predicts wiki 节点 + 相关节点

$ lark-cli drive +search --query "POST predicts"
# total: 10 — method + path 都正确匹配，含 highlighted snippet

$ lark-cli drive +search --query "创建预测"
# total: ≥3 — 中文 summary 命中，含 zombie + live 节点

$ lark-cli drive +search --query "isFirstPredict"
# total: 2 — substring 匹配到 predict（不到字段名级别，但够用）

返回结果含 summary_highlighted（带 <h>...</h> 标记的命中片段）+ result_meta.url / token / edit_user / timestamps。对 AI 而言完全足够搜索用。

注意：--doc-types wiki filter 会让结果全 0 — 不要 set 这个 flag（这是 lark-cli bug 或不熟知用法）。直接搜不带 filter，让 doc + wiki + docx 都参与召回，再用 entity_type 在客户端筛。

4.2 验证 2：AI 直接读 spec 文件就够

对当前实际使用方 forecast-market-api：

指标值对 AI 来说
spec 文件api/openapi.snapshot.json就在 repo 里，AI 直接 Read 没障碍
大小84KB / 3512 行一次读完远低于上下文限制
paths / endpoints34 paths / 40 endpoints规模小，整体 grep 都行
components.schemas61 个$ref 追链是 AI 基本能力，不构成痛点
$ref 总出现115 次POST /predicts 处理 3 个 ref 就到底，浅

结论：本项目场景下，AI Read api/openapi.snapshot.json + 追 $ref 完全足够回答接口问题。manifest 的"省 $ref 追链"价值在小项目里不成立。

4.3 价值矩阵 — manifest / recall 何时才值得做

场景spec 大小是否需要 manifest
forecast-market-api（当前）40 endpoints / 84KB不需要 — 直接读 spec
中型项目200 endpoints / 500KB边际 — AI 仍能读完但 token 浪费多
大型项目1000+ endpoints / 5MB+需要 — AI 没法一次读完
跨 service 联邦≥3 个 service 并存需要 — 跨 spec 检索单靠 Read 不行

当前 forecast-market-api 落在第一行，没有任何一个使用方达到 manifest 的回报门槛。等真出现 200+ endpoints 项目或跨 service 联邦需求时，再回来读这份文档启动实现。

5. 实际落地的替代方案（已做）

openapi-lark init 命令新增能力：在 --config 所在目录写 .openapi-lark.yaml 时，同时往 CLAUDE.md 注入一段 openapi-lark 管理块，告诉 AI 怎么读 spec 和怎么搜 wiki：

<!-- openapi-lark:claude-md:begin -->
## API spec (openapi-lark 自动注入；本块被 openapi-lark init 维护)

本项目接口文档已通过 openapi-lark 同步到飞书 wiki（service: `forecast-market-api`）：

- OpenAPI 源：`api/openapi.snapshot.json`（本地文件，直接 Read 就能拿到 paths + components.schemas）
- 飞书 wiki 镜像：https://portwind.jp.larksuite.com/wiki/...

### AI 查接口的两条路

- 直接 Read `api/openapi.snapshot.json`：有 `paths` + `components.schemas` 双轨，遇到 `$ref` 跟过去就行
- 搜飞书 wiki：`lark-cli drive +search --query "<关键字>"`（path / 中文 summary / 单词都能命中，返回带 highlighted snippet）
- 拿单个文档全文：`lark-cli docs +fetch --doc <token> --api-version v2`

### spec 改了之后

- 重新 sync 推到飞书：`openapi-lark sync`
- 加 git hook 自动同步：`openapi-lark install-hook`
<!-- openapi-lark:claude-md:end -->

特性：

  • 幂等 — begin/end marker 之间的内容会被替换；marker 之外的用户内容保留不动
  • opt-out — openapi-lark init --no-claude-md 跳过注入
  • 已实现 + 已测试 — buildClaudeMdSection / injectClaudeMd 函数 + 9 个测试（create / append / replace / preserve user content / idempotent）
  • URL 型 openapi 自动切提示 — 提示改为 curl 而不是 Read

6. 反思 — 为什么走了 5 轮才发现前提错

  1. 第一轮没验证 lark-cli 能力 — 直觉外推说"飞书 search 精度差"，没跑一条命令拿数据，整个 v1-v3 spec 都建在这个未经验证的前提上
  2. 没跟基线对比 — 一直在比较 "manifest 方案 vs MCP 方案"，没问 "AI 直接 Read spec 文件 vs manifest" 哪个更轻
  3. codex 评审在抓表面一致性 — 5 轮 codex 检查了 spec 内部矛盾（命令名 / 路径 / scope），但没人挑战"这功能真的需要做吗"
  4. 修复：今后任何针对 lark-cli 能力的判断必须先跑命令拿数据（已写到 memory feedback_verify_before_dismissing_lark_cli）。新 feature 设计第一步必须验证"无所作为时基线体验有多差"

7. 何时回来重新考虑这套设计

  • 项目接入 openapi-lark 的 spec 突破 ~300KB（AI 读 token 成本可感）
  • 用户 .openapi-lark.yaml 配 ≥3 个 service 且明确想跨 service 查
  • 飞书 search 政策变化或 v2 接口废弃
  • AI 客户端从 Claude Code 扩展到不能跑 Bash 的环境（这时回来考虑的应该是另一份新设计）
