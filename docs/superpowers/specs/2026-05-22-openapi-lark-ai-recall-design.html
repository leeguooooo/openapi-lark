1. 问题与目标

openapi-lark v0.2.x 解决的是「项目 OpenAPI spec → 飞书 wiki 人可读文档」。但 AI 读不到这些文档：

  • 飞书 wiki 的 markdown 渲染版（参数表 / 响应表 / JSON 示例）是给人类视觉设计的，AI 解析时容易丢字段
  • 没有 endpoint 索引 — AI 不知道「forecast-market-api 里有没有 predicts 接口」该去哪查
  • 没有结构化 schema 出口 — AI 写 client 代码时要从 markdown table 反推 TypeScript interface，错误率高
  • 飞书 wiki 自带搜索精度差，不适合按 path / method / schema 字段名精准查

目标：每次 sync 之后产出一份机器可读的 manifest.json，包含 dereferenced schema、endpoint 元数据、本地 md 文件路径、飞书 wiki URL。新增 recall 子命令组让 AI（在 Claude Code 里通过 Skill 触发）能查询这份 manifest。

2. 范围裁定

brainstorm 过程中考虑过三种架构：

方案形态v0.3 取舍
A. CLI 子命令 + Skill 触发（本案）openapi-lark 长出 recall 子命令组，输出 JSON，AI 通过 Skill 触发 spawn 子进程采用 上下文 / 内存零额外开销，跟既有 skill 分发形态一致
B. stdio MCP server（sibling skill）新开 openapi-lark-recall repo，跑 @modelcontextprotocol/sdk stdio server否决 用户拒绝（见下方决策记录）；tool schemas 永驻会话上下文 + 进程保活 + 跨 repo manifest schema 漂移
C. 飞书 wiki 搜索桥纯翻译层调 drive +search（Search v2）否决 飞书搜索精度太差，无结构化输出
MCP pivot 决策记录（2026-05-22）：v1 spec 原本是方案 B（MCP server + sibling repo）。用户在 codex 复审后明确表态「我就不想做 mcp」。codex 反驳「未来 Cursor / Trae 客户端会用不上 CLI 只能用 MCP」，但用户接受这个锁定，理由是：上下文 / 内存成本不愿付，且 openapi-lark 已是 npx skill 分发形态，与 CLI + Skill 触发完全契合。v0.3 不出 MCP 方案；未来若有强烈多 client 需求再单独提案。

3. 用户 / AI 交互形态

角色怎么用
开发者本人直接 openapi-lark recall search 预测，shell 输出 JSON 或 pretty 模式（--pretty）
Claude Code 里的 AI用户问「forecast-market-api 的预测接口怎么用」→ 触发 openapi-lark skill 的 recall 路径 → AI 用 Bash 工具跑 openapi-lark recall get POST /api/v1/predicts --service forecast-market-api → 拿 JSON 解析
CI / 脚本同开发者，stable JSON 输出便于 jq pipeline

SKILL.md 触发词：「查接口 / 找 endpoint / openapi recall / 接口 spec / 这个 API 怎么用」。description 里列出 4 个子命令的用途，AI 看 description 自助选择该跑哪条。

4. manifest.json schema

每次 sync（endpoint 模式）末尾自动写到 .openapi-lark/manifest.json，与 sync-lock.json 同期产出。这是整个 v0.3 的灵魂 — 所有 recall 子命令都消费它。

{
  "version": 1,
  "generatedAt": "2026-05-22T01:30:00Z",
  "tool": { "name": "openapi-lark", "version": "0.3.0" },
  "services": {
    "forecast-market-api": {
      "specPath": "api/openapi.yaml",      // 本地: 相对 basedir; URL: 原始 URL
      "specSha256": "abc123...",            // resolved + dereferenced spec 的 sha
      "wikiParent": {
        "docToken": "PARENT_TOKEN",
        "spaceId": "7641...",
        "nodeToken": "NODE_PARENT",
        "title": "API Docs Parent",
        "url": "https://portwind.jp.larksuite.com/wiki/NODE_PARENT"
      },
      "endpoints": [
        {
          "method": "POST",
          "path": "/api/v1/predicts",
          "operationId": "createPredict",
          "tagId": "games",
          "groupKey": "predicts",
          "summary": "创建预测（下注）",
          "descriptionFirstLine": "用户对某场比赛下注，扣除余额并记录预测",
          "deprecated": false,
          "internal": false,
          "wiki": {
            "nodeToken": "MnaidT2w...",
            "objToken": "DrW7d...",
            "url": "https://portwind.jp.larksuite.com/wiki/MnaidT2w..."
          },
          "localMdPath": ".openapi-lark/forecast-market-api/games/predicts/创建预测_POST_api_v1_predicts.md",
          "requestBodyShape": ["amount", "predictId", "side"],
          "responseShape": { "200": ["isFirstPredict", "predictId", "balance"], "400": ["error", "code"] },
          "requestBodySchema": { /* full dereferenced JSON Schema */ },
          "responseSchemas": { "200": { /* ... */ }, "400": { /* ... */ } },
          "syncStatus": "ok",
          "lastSuccessfulSyncAt": "2026-05-22T01:30:00Z"
        }
      ]
    }
  }
}
requestBodyShape / responseShape 是关键创新：把 schema 顶层字段名平铺到搜索语料里。这样 AI 问 "isFirstPredict 是哪个接口返回的" 时能直接命中，正是触发本次设计的 bug 场景。
大小预估：单 endpoint 含完整 dereferenced schema 可能 5–30KB。200 endpoints 的 manifest.json 估计 1–5MB。不分文件，单条 search / get 命令冷启动加载一次（≈ 50–200ms），不持续驻留。如果未来 manifest 突破 10MB 再考虑分片。

字段语义细则

字段说明
services.<svc>.specPath本地文件时是相对 basedir 的路径；URL 型 openapi 时直接存原始 URL（含 scheme）
services.<svc>.specSha256始终是 resolved + dereferenced spec JSON 的 sha256（不是源文件的 sha）。URL 和本地文件统一这条规则
endpoints[].deprecated来自 OpenAPI 标准字段 operation.deprecated
endpoints[].internal来自 OpenAPI 扩展 x-internal: true。无该扩展时默认 false
endpoints[].descriptionFirstLineoperation.description 第一行（trim，最长 200 字符）
endpoints[].wiki.url用 sync-lock 里记的 host + nodeToken 拼
endpoints[].localMdPath相对 basedir，跟 sync-endpoint.ts 里 outRel 同一逻辑（含 .openapi-lark/ 前缀）
endpoints[].requestBodyShaperequestBody schema 顶层 properties 的 key 数组；不递归。无 requestBody 时空数组
endpoints[].responseShape.<status>同上，按状态码（200/201/400/...）分别给出 shape
endpoints[].syncStatus"ok" / "failed" — 该 endpoint 本次 sync 是否成功推到飞书。failed 时本地 md 通常仍有效
endpoints[].lastSuccessfulSyncAt最近一次 syncStatus: "ok" 的时间戳。failed 时保留上次的值，便于判断陈旧度

v1 forward compatibility

  • v1 期间：只允许加字段，不允许删 / 改语义。recall 命令遇到未知字段一律忽略
  • 删字段或语义变更：必须 bump 到 v2，并在 spec 单开 migration note
  • recall 启动时强校验 version 字段；version mismatch → 退出非零，stderr 指明 regen 命令

5. recall 子命令接口

所有 recall 子命令默认输出 JSON（machine-readable）。--pretty 切换到 table / formatted 给人看。所有命令冷启动时读一次 manifest.json（加载 + 校验 ≈ 50–200ms），完事进程退出。

manifest 发现路径

每个子命令按以下优先级解析 manifest 位置：

  1. --manifest <path> 显式 flag（最高）
  2. 环境变量 OPENAPI_LARK_MANIFEST
  3. 从当前工作目录 (process.cwd()) 向上查找最近的 .openapi-lark/manifest.json（停在 git root 或文件系统根）
  4. 找不到 → exit 2 + stderr 提示

向上查找逻辑跟现有 config/load.ts 的 .openapi-lark.yaml 解析一致，复用同一 helper。

子命令用途关键参数
openapi-lark recall list-services 列所有 service 及 endpoint 数 —
openapi-lark recall list-endpoints 列指定 service 的所有 endpoint（method + path + summary + wikiUrl + deprecated） --service <name>，可选 --tag <id>、--include-deprecated
openapi-lark recall search <query> 按关键字搜，返回相关性排序的命中数组 --service、--method、--tag、--limit 10
openapi-lark recall get <METHOD> <path> 返回单个 endpoint 完整数据：metadata + dereferenced JSON Schema + 渲染好的 markdown 内容（从 localMdPath 读） --service <name>（必填），--live（走 lark-cli 拉飞书最新 markdown 覆盖本地）

输出 schema — get

返回 manifest 中该 endpoint 的全部字段，再附加 markdownContent（从 localMdPath 读出的本地 md 内容）。如果 localMdPath 文件不存在，markdownContent: null + missingReason 字段。

$ openapi-lark recall get POST /api/v1/predicts --service forecast-market-api
{
  "service": "forecast-market-api",
  "method": "POST",
  "path": "/api/v1/predicts",
  "operationId": "createPredict",
  "tagId": "games",
  "groupKey": "predicts",
  "summary": "创建预测（下注）",
  "descriptionFirstLine": "用户对某场比赛下注...",
  "deprecated": false,
  "internal": false,
  "wiki": { "nodeToken": "MnaidT2w...", "objToken": "DrW7d...", "url": "..." },
  "localMdPath": ".openapi-lark/forecast-market-api/games/predicts/创建预测_POST_api_v1_predicts.md",
  "requestBodyShape": ["amount", "predictId", "side"],
  "responseShape": { "200": ["isFirstPredict", "predictId", "balance"] },
  "requestBodySchema": { /* full dereferenced JSON Schema */ },
  "responseSchemas": { "200": { ... }, "400": { ... } },
  "markdownContent": "# 创建预测...\n\n## 请求参数\n...",
  "syncStatus": "ok",
  "lastSuccessfulSyncAt": "2026-05-22T01:30:00Z"
}
markdownContent 是 recall get 运行时从 localMdPath 读取的派生字段，不存在 manifest.json 里（避免数据冗余 + 漂移）。--live 模式下用 lark-cli docs +fetch 拉到的内容覆盖此字段；live 失败时静默降级回读 localMdPath。

6. lark-cli 边界（--live）

原则：凡是要碰飞书 API，100% 走 lark-cli。recall 默认完全本地（不调 lark-cli），--live 走 lark-cli 拉最新。

场景是否走 lark-cli命令
默认 search / list / get（从 manifest）否—
get --live 拉最新 docx是lark-cli docs +fetch --doc <objToken> --api-version v2（lark-doc v2 skill 返回 Lark-flavored markdown）
--live 失败处理—静默降级回读 localMdPath 本地 md 文件作为 markdownContent，stderr warn 一次
auth 前置条件：--live 模式要求 lark-cli 已 auth login，且应用拥有 wiki:wiki:readonly + docx:document:readonly + drive:drive:readonly scope。recall 在启用 --live 时先跑 lark-cli auth status 自检，未登录或 keychain not initialized 则 stderr 提示并禁用 live 本次（仍能跑本地查询）。
v0.3 不做 search --live：飞书 drive +search 精度差（method / path 命中弱），即使叠加也帮助有限，且会让 AI 搜索体验变慢。get --live 才有真实价值（拉浏览器手改过的最新 docx）。

7. sync 端的改动（最小侵入）

  1. 新增 src/manifest.ts — 纯函数 buildManifest(input): Manifest，输入 dereferenced API + sync-lock + node-map + endpoint slices，输出 Manifest 对象
  2. 修改 src/commands/sync-endpoint.ts：在 runEndpointSync 内声明 const manifestEntries: ManifestEndpointEntry[] = []；pushEndpointLeaf 内部往里 push 一条（即使 push 失败也 push，failed 标 syncStatus: "failed"，lastSuccessfulSyncAt 沿用 lock 里上一次）；末尾循环全部 service 结束后调 writeManifest()
  3. 写入时机：sync 全部 service 都跑完后写一次。失败容忍：哪怕部分 leaf push 失败，只要本地 md 已渲染、metadata 齐全，manifest 仍写入（让 recall 至少能服务部分内容）。dry-run 时跳过写入
  4. 新增 src/commands/manifest.ts + openapi-lark manifest 命令：不 sync、不推飞书，只重生 manifest（spec 改了但还没 sync 时用）
  5. CHANGELOG 加段：每次 sync 末尾写 manifest.json；新增 recall 子命令组

9. 关键词搜索 + 排序

不引入 BM25 / TF-IDF / 向量嵌入。简单加权够用。

分词规则

  • 大小写：折叠到小写后比较
  • 路径分隔符：path 被 / 切分，例如 /api/v1/predicts → ["api", "v1", "predicts"]
  • camelCase：字段名按 camelCase 拆词，isFirstPredict → ["is", "first", "predict", "isfirstpredict"]（同时保留原 token，避免精确匹配丢失）
  • 中英文：query 中文按整段匹配（不切字），如 "创建预测" 直接在 summary / descriptionFirstLine 里查 substring；英文按空白 + 标点切分
  • 停用词：v0.3 不做停用词过滤（语料小，无明显噪声）
score(endpoint, queryTokens) =
    3 * count(token ∈ path)
  + 2 * count(token ∈ summary)
  + 2 * count(token ∈ operationId)
  + 1 * count(token ∈ descriptionFirstLine)
  + 1 * count(token ∈ requestBodyShape ∪ responseShape.<all_status>)
  + 1 * count(token ∈ tagId)

filter: --method? --tag? --service? 命中过滤 + deprecated 默认排除
sort:   score desc, ties 按 path asc
limit:  默认 10, --limit 上限 50

实测 200 endpoint 关键词搜在 V8 上 < 5ms（manifest 加载本身 50-200ms，是主成本）。

10. 错误处理 / 退出码

失败模式退出码行为
正常0stdout JSON
manifest.json 不存在2 (EXIT_CONFIG)stderr 提示先跑 sync 或 manifest
manifest version mismatch2stderr 指明 regen
manifest JSON Schema 验证失败2stderr 打印首个错误字段
get 找不到 method+path3 (EXIT_NOT_FOUND)stderr 提示可能的近似命中（fuzzy hint）
get 命中但 localMdPath 不存在0stdout JSON 但 markdownContent: null + missingReason: "local md not found, run sync"
--live 调 lark-cli 超时0stderr warn，降级本地
--live 鉴权失败（含 keychain not initialized）0stderr warn 一次，本次禁用 live，输出本地结果
search 无命中0stdout 空 hits 数组，total=0
语法错误（参数缺失等）1 (EXIT_BUSINESS)stderr usage 提示

核心原则：能给部分结果时返回 0 + JSON 标记，让 AI 拿到结构化降级信息；只有完全没法工作时才非 0。

13. 风险与未解

  • 每次 spawn 子进程的冷启动成本：50–200ms 解析 + 加载 manifest。AI 单轮可能调多次（先 list-services 再 search 再 get），累加 ~500ms 体感可接受，但非 MCP 的代价就是它。如果未来真痛，再加 recall serve 选项暴露 HTTP 接口本地复用（不是 MCP）。
  • AI 自助发现不如 MCP 直接：CLI 模式下 AI 要先看 SKILL.md 才知道有 recall 子命令。SKILL.md description 写得好坏直接影响触发准确率，需要 v0.3 ship 前 dogfood 验证。
  • flag 变更破坏 AI 调用链：CLI flag 改了 AI 旧 prompt 还在用就会失败。对策：v1 期间 flag 只加不删；deprecated flag 至少保留两个 minor 版本兼容。
  • manifest 大小爆炸：dereferenced schema 递归型可能很大。预案：观察实际项目（forecast-market-api ~50 endpoints 估计 < 2MB），超 10MB 再加 schema 引用计数 / 拆文件。
  • 用户实际场景验证：v0.3 发布后需要在 forecast-market-api 真用一周，观察 AI 是否真的用上了 recall 子命令，再决定 v0.4 优先级。
  • 锁定 Claude Code：codex 指出 Cursor / Trae 这类客户端不一定能自动 spawn CLI。用户接受这个锁定（详 §2 决策）。如果将来真有 Cursor 用户提需求，再考虑加 MCP wrapper 入口。
