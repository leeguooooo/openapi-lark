1. 问题与目标

openapi-lark v0.2.x 解决的是「项目 OpenAPI spec → 飞书 wiki 人可读文档」。但 AI 读不到这些文档：

  • 飞书 wiki 的 markdown 渲染版（参数表 / 响应表 / JSON 示例）是为人类视觉设计的，AI 解析时丢字段
  • 没有 endpoint 索引 — AI 不知道「forecast-market-api 里有没有 predicts 接口」该去哪查
  • 没有结构化 schema 出口 — AI 写 client 代码时要从 markdown table 反推 TypeScript interface，错误率高
  • 飞书 wiki 自带搜索精度差，不适合按 path / method / schema 字段名精准查

目标：每次 sync 之后产出一份机器可读的 manifest.json，包含 dereferenced schema、endpoint 元数据、本地 md 文件路径、飞书 wiki URL。再用一个 stdio MCP server 把它喂给 Claude Code / Cursor / 任意 MCP 客户端。

2. 范围裁定

brainstorm 过程中考虑了三种架构：

方案形态v0.3 取舍
A. 本地单项目 stdio MCP读 .openapi-lark/manifest.json + 本地 md采用 ROI 最高，1-2 天能落地
B. 跨项目中心索引共享后端 + 多项目 manifest 汇总v0.4+ 要部署后端，门槛高，但 A 是它的必经路径
C. 飞书 wiki 搜索桥纯翻译层调 wiki +node-search否决 飞书搜索精度太差，无结构化输出
用户场景判定：开发者在 IDE 里写代码时让 AI 自助查自己项目的接口。这正是 A 的射程范围。"团队群里 @bot 问接口" 是 B/C 场景，不在 v0.3 范围内。

3. 分两个 skill 的理由

sync 和 recall 生命周期、触发词、消费者完全不同，强行揉一个 SKILL.md 会让 AI 派发不准。

维度openapi-lark（已存在，保持）openapi-lark-recall（新）
角色OpenAPI → 飞书 wiki 文档树本地 manifest + 飞书 wiki → AI MCP
触发场景CI / git hook / 一次性 buildIDE 长驻 stdio MCP server
命令sync / lint / render / doctor / init / install-hookmcp / manifest（regen）
跟 lark-cli 关系同步推内容，深度依赖仅 --live 选项走 lark-cli
SKILL.md 触发词"同步接口文档到飞书 / openapi 推到飞书""AI 查我们接口 / 接口 MCP / openapi recall"
单装可能否（核心）是 — 团队消费方只装 recall 也能用别人 sync 好的 manifest
跨 skill 契约manifest.json schema v1（见 §5）

4. 仓库 / 发布形态

决定：两个独立 repo，不做 monorepo。理由：

  • openapi-lark 已发布、有自己的 release pipeline + CHANGELOG + git 历史；为加 sibling 而把它降级到子目录代价大
  • monorepo 重构有可能破坏现有的 npx -y -p "github:leeguooooo/openapi-lark#main" 安装路径，v0.3 不值这风险
  • 共享只是一份小 JSON Schema，不值得引入 workspace 管理
  • 未来真要合 monorepo，"合两个 repo" 比 "拆一个 repo" 容易
仓库npm 包skill 装法引用方式
leeguooooo/openapi-lark（既有）openapi-lark（或 GitHub 源）npx skills add github:leeguooooo/openapi-lark—
leeguooooo/openapi-lark-recall（新建）openapi-lark-recallnpx skills add github:leeguooooo/openapi-lark-recall—

manifest schema 的 single source of truth：放在 openapi-lark repo 的 schemas/manifest.v1.schema.json（JSON Schema 标准）。recall repo 在自己代码里 copy 一份验证用的 JSON Schema（不依赖第三方包），版本号字段强校验。两边 TS 类型可以各自手写，因为 manifest 字段少且不易变。

5. manifest.json schema

每次 sync（endpoint 模式）末尾自动写到 .openapi-lark/manifest.json，与 sync-lock.json 同期产出。

{
  "version": 1,
  "generatedAt": "2026-05-22T01:30:00Z",
  "tool": { "name": "openapi-lark", "version": "0.3.0" },
  "services": {
    "forecast-market-api": {
      "specPath": "api/openapi.yaml",      // 相对 basedir
      "specSha256": "abc123...",            // 用于失效检测
      "wikiParent": {
        "docToken": "PARENT_TOKEN",
        "spaceId": "7641...",
        "nodeToken": "NODE_PARENT",
        "title": "API Docs Parent",
        "url": "https://portwind.jp.larksuite.com/wiki/NODE_PARENT"
      },
      "endpoints": [
        {
          "method": "POST",
          "path": "/api/v1/predicts",
          "operationId": "createPredict",
          "tagId": "games",
          "groupKey": "predicts",
          "summary": "创建预测（下注）",
          "descriptionFirstLine": "用户对某场比赛下注，扣除余额并记录预测",
          "deprecated": false,
          "internal": false,
          "wiki": {
            "nodeToken": "MnaidT2w...",
            "objToken": "DrW7d...",
            "url": "https://portwind.jp.larksuite.com/wiki/MnaidT2w..."
          },
          "localMdPath": "forecast-market-api/games/predicts/创建预测_POST_api_v1_predicts.md",
          "requestBodyShape": ["amount", "predictId", "side"],
          "responseShape": { "200": ["isFirstPredict", "predictId", "balance"], "400": ["error", "code"] },
          "requestBodySchema": { /* full dereferenced JSON Schema */ },
          "responseSchemas": { "200": { /* ... */ }, "400": { /* ... */ } }
        }
      ]
    }
  }
}
requestBodyShape / responseShape 是关键创新：把 schema 顶层字段名平铺到搜索语料里。这样 AI 问 "isFirstPredict 是哪个接口返回的" 时能直接命中，正是触发本次设计的 bug 场景。
大小预估：单 endpoint 含完整 dereferenced schema 可能 5–30KB。200 endpoints 的 manifest.json 估计 1–5MB。不分文件，MCP server 一次性载入；如果未来突破 10MB 再考虑分片。

6. MCP tool 接口

命名规范：snake_case（MCP 生态主流）。零冲突于 lark-openapi-mcp 的 im.v1.* 命名空间。

ToolInputOutputLive 模式 (--live)
list_services — [{ name, endpointCount, specPath, specSha256 }] 同 default（manifest 是权威）
list_endpoints { service?, tag?, deprecated? } [{ method, path, summary, tagId, wikiUrl, deprecated }] 同 default
search_endpoints { query, service?, method?, tag?, limit=10 } 按相关性排序的命中数组（见 §10） 同 default + lark-cli wiki +node-search --query <q> 命中并集
get_endpoint { service, method, path } { ...metadata, requestBodySchema, responseSchemas, markdownContent } 同 default，但 markdownContent 走 lark-cli docs +fetch --doc <objToken> 拉飞书最新

get_endpoint 返回结构化 JSON Schema + 渲染 markdown 双轨。AI 写 TypeScript 客户端时拿 schema，给人复述时拿 markdown。把"AI 可读"做到底。

7. lark-cli 使用边界

原则：凡是要碰飞书 API，100% 走 lark-cli，不自己写 HTTP / OAuth。recall skill 默认完全本地，--live 走 lark-cli。

场景是否走 lark-cli命令
默认 search / get（从 manifest）否—
--live search 兜底是lark-cli wiki +node-search --query "<q>" --space-id <sid>
--live get 拉最新 docx是lark-cli docs +fetch --doc <objToken> --api-version v2 --doc-format markdown
--live 失败处理—静默降级到本地 manifest，stderr warning 一次

8. sync skill 端的改动（最小侵入）

  1. 新增 src/manifest.ts — 纯函数，输入 dereferenced API + sync-lock + node-map，输出 Manifest 对象。单测好写。
  2. 修改 src/commands/sync-endpoint.ts runEndpointSync 末尾：累积所有 endpoint 的 metadata + schema，在 sync 完成且非 dryRun 时调用 writeManifest()。
  3. 新增 src/commands/manifest.ts + openapi-lark manifest 命令：不 sync、不推飞书，只重生 manifest（用 sync-lock + node-map 里已有的 nodeToken/url 信息，加上重新 dereference 的 spec）。useful for: spec 改了但还没 sync，本地想先让 AI 用新 manifest。
  4. 新增 schemas/manifest.v1.schema.json — JSON Schema 文件，给 recall skill 验证用。
  5. CHANGELOG 加一段：每次 sync 末尾会写 manifest.json；可被 openapi-lark-recall 消费。

10. 关键词搜索 + 排序

不引入 BM25 / TF-IDF / 向量嵌入。简单加权够用。

score(endpoint, queryTokens) =
    3 * count(token ∈ path)
  + 2 * count(token ∈ summary)
  + 2 * count(token ∈ operationId)
  + 1 * count(token ∈ descriptionFirstLine)
  + 1 * count(token ∈ requestBodyShape ∪ responseShape)
  + 1 * count(token ∈ tagId)

filter: method? tag? service? 命中过滤
sort:   score desc
limit:  默认 10, 最多 50

实测 200 endpoint 关键词搜在 V8 上 < 5ms，完全足够 MCP tool roundtrip 体验。

      "cwd": "/Users/leo/tk.com/forecast-market-api"
    }
  }
}

Cursor / Trae（同上格式）

用 cwd 让 recall 知道去哪找 .openapi-lark/manifest.json。多项目挂多个 entry，工具名通过 server 名（forecast-api-recall / admin-api-recall）区分。

启用 live 模式

"args": ["-y", "openapi-lark-recall", "mcp", "--live"]

此时 search/get 会用 lark-cli 兜底，需要 lark-cli 已登录并有 wiki:node:retrieve + docx:document:readonly scope。

12. 错误处理 / 版本契约

失败模式recall 行为
manifest.json 不存在退出非零；提示用户先跑 openapi-lark sync 或 openapi-lark manifest
manifest version mismatch退出非零；提示升级 recall 或 regen manifest
manifest JSON Schema 验证失败退出非零；打印第一个错误字段
get_endpoint 找不到 method+path返回 MCP error（不抛进程），AI 拿到 "endpoint not found"
--live 调 lark-cli 超时stderr warn，降级到本地 manifest
--live 调 lark-cli 鉴权失败stderr warn 一次，本次进程后续不再尝试 live

13. 测试策略

  • sync skill：test/manifest.test.ts — manifest 生成单测，输入构造的 dereferenced API + sync-lock + node-map，断言 manifest 各字段
  • recall skill / search：test/search.test.ts — 排序加权单测，构造 manifest，验证 query 返回顺序
  • recall skill / schema 校验：test/manifest-load.test.ts — 各种坏 manifest（缺字段 / 版本错 / json 坏）触发预期错误
  • recall skill / MCP roundtrip：test/mcp-roundtrip.test.ts — 起 stdio server，从 stdin 发 list_tools / call_tool，断言 response。每个 tool 一个 happy-path
  • recall skill / live mode：test/live-fallback.test.ts — fake lark-cli 模拟超时 / 鉴权失败，验证降级

14. 明确不做（v0.3）

项原因
跨项目联邦 / 中心索引v0.4+，先把单项目场景验证好
语义搜索 / embeddings< 500 endpoint 关键词够用，避免引入 sqlite / sentence-transformers 重依赖
generate_code_sample(method, path, lang)v0.4 候选；要 widdershins 之外的 codegen，工程量大
HTTP / SSE transportstdio 够用，HTTP 等真有跨机器需求
鉴权 / token gate本地 stdio 无需，跨网络部署再说
飞书 wiki 反向写回 manifest方向反了；manifest 是 sync 输出，不是输入
mode: tree 支持tree 模式没有 leaf 级 md，AI recall 意义小；只覆盖 mode: endpoint
endpoint similar / relatedv0.4 候选

15. 风险与未解

  • manifest 大小爆炸：dereferenced schema 递归型可能很大。预案：观察实际项目（forecast-market-api ~50 endpoints 估计 < 2MB），超 10MB 再加 schema 引用计数 / 拆文件。
  • multi-tenant 项目：一个仓库多个 service 时 cwd 仅指 repo root 没问题，server 端按 service 切片。
  • recall skill 跟 openapi-lark 的版本兼容：靠 manifest schema version 字段强校验。版本 1 期间不破坏字段，加字段允许（recall 忽略未知）。
  • lark-cli 升级破坏 live 模式：live 是兜底不是核心，破了静默降级，CI 不会因此挂。
  • schemas/manifest.v1.schema.json 跨 repo 漂移：每次 openapi-lark 改 manifest 时手动 copy 到 recall repo，并在 recall release notes 写清。后续若漂移频繁，再考虑发 @leeguoooo/openapi-lark-manifest-schema npm 包。
  • 用户实际场景验证：v0.3 发布后需要在 forecast-market-api 真用一周，观察 AI 是否真的用上了 search / get，再决定 v0.4 优先级。
