# openapi-lark 接入教程

> 把任意 OpenAPI 项目的接口文档同步到飞书 wiki，每个接口一个独立子节点。
>
> 受众：第一次用本工具，准备接入自己的项目。

---

## 这工具到底做什么

你给它：
- 一份 OpenAPI 3.x yaml / json（绝大多数后端都有，比如 `api/openapi.yaml`）
- 一个飞书 wiki 父节点 URL

它给你：
- 飞书 wiki 树：父节点下按 tag 分组，每个接口一个独立子文档
- 自动按中文 summary 命名（`获取用户信息 — GET /api/user`）
- 完整的参数表 / 响应字段表 / JSON 示例
- 改一行 openapi 重跑只推一个 doc（hash 缓存）

✗ 不做：手写文档管理 / 多人协作评论（被同步的 docx 是只读展示）

## 准备工作（5 分钟）

### 0.1 装 lark-cli

```bash
brew install lark-cli       # macOS
# 或 https://github.com/larksuite/lark-cli 看其他平台
```

### 0.2 登录

```bash
lark-cli auth login
# 浏览器打开 verification URL，授权完会自动跳回
```

如果之后报 `missing required scope(s): wiki:node:write`，加 scope 重登：

```bash
lark-cli auth login --scope "wiki:node:write wiki:node:move"
```

### 0.3 验证

```bash
lark-cli --version          # 应该 >= 0.1.0
lark-cli auth status        # 应该显示已登录
```

### 0.4 安装本 skill（可选）

如果你用 Claude Code，让 Claude 知道这个工具：

```bash
npx skills add leeguooooo/openapi-lark
```

不用 skill 也行，所有命令都能直接 `npx -y -p github:leeguooooo/openapi-lark openapi-lark <cmd>` 调。

## 接入步骤（10 分钟）

### Step 1 — 在飞书 wiki 建一个父节点

打开你团队的 wiki space → 右键空白处 / 顶部「+」→ 新建 docx → 标题随便起（比如「项目 X API」）→ **不要往里写任何东西**。

复制这个新节点的 URL，形如：

```
https://xxx.larksuite.com/wiki/Uc6hwkuaxihmXVkqcqejKpiDpTf?fromScene=spaceOverview
```

**⚠️ 警告**：不要复用已经有内容的 wiki 节点。本工具会清空它的内容并自己接管。

### Step 2 — init

进到你项目根目录（含 openapi.yaml 的目录），跑：

```bash
npx -y -p github:leeguooooo/openapi-lark openapi-lark init \
  --name my-service \
  --openapi api/openapi.yaml \
  --doc-url 'https://xxx.larksuite.com/wiki/Uc6hwkuaxihmXVkqcqejKpiDpTf'
```

完成后会生成 `.openapi-lark.yaml`，类似：

```yaml
engines:
  larkCli: ">=0.1.0"
services:
  - name: my-service
    openapi: api/openapi.yaml
    docToken: Uc6hwkuaxihmXVkqcqejKpiDpTf
```

### Step 3 — 编辑 `.openapi-lark.yaml`

至少加 `mode` 和 `parentTitle`：

```yaml
engines:
  larkCli: ">=0.1.0"
maxPushBytes: 1000000              # 推荐设到 1MB，给点余量

services:
  - name: my-service
    openapi: api/openapi.yaml
    docToken: Uc6hwkuaxihmXVkqcqejKpiDpTf
    mode: endpoint                  # ← 加：每接口一个 wiki 子节点
    parentTitle: 项目 X              # ← 加：锁定父节点标题
    # 可选：把 openapi tag 翻译成中文显示名
    tagAliases:
      user: 用户接口
      order: 订单接口
      admin: 管理端接口
```

把 `.openapi-lark.yaml` 提交进 git。`.gitignore` 加一行：

```
.openapi-lark/
```

`.openapi-lark/` 是本地渲染缓存 + 同步 lockfile，不需要进 git。

### Step 4 — 跑第一次同步

```bash
npx -y -p github:leeguooooo/openapi-lark openapi-lark sync --parallel 4
```

工具会：
1. 读 `api/openapi.yaml`
2. 按 tag 切成 N 组，每组按接口再切成 M 个独立 sub-api
3. 在飞书父节点下创建 tag 中间节点
4. 在每个 tag 中间节点下创建接口叶子节点
5. 给每个节点 push 渲染好的 md

耗时跟接口数线性相关：

| 接口数 | 估时 |
|---|---|
| 50 | 3-5 min |
| 100 | 5-10 min |
| 200 | 10-16 min |

期间会打印 `[sync] ✓ my-service :: user :: 获取用户信息 — GET /api/user (1.8s)` 这样的进度。

### Step 5 — 看效果

回到飞书 wiki，刷新页面，你应该看到：

```
项目 X
├── 用户接口
│   ├── 获取用户信息 — GET /api/user
│   ├── 创建用户 — POST /api/user
│   ├── ...
├── 订单接口
│   ├── ...
└── 管理端接口
    └── ...
```

点开任意一个接口叶子，应该看到：

```
[飞书 docx 标题: 获取用户信息 — GET /api/user]

`GET /api/user`

获取用户详细信息

### 参数
| 名称 | 位置 | 类型 | 必填 | 描述 |
| userId | query | string | true | 用户 ID |

### 响应
| 状态码 | 含义 | 描述 | Schema |
| 200 | OK | 用户信息 | Inline |
| 404 | Not Found | 用户不存在 | Inline |

### 响应 Schema
Status Code 200
| 名称 | 类型 | 必填 | 约束 | 描述 |
| » success | boolean | true | none | 请求是否成功 |
| » data | object | false | none | 响应数据 |
| »» id | string | false | none | 用户 ID |
| »» name | string | false | none | 用户名 |

### 响应示例 (200)
```json
{
  "success": true,
  "data": {
    "id": "string",
    "name": "string"
  }
}
```
```

## 日常使用

### 改了 openapi 重新同步

```bash
npx -y -p github:leeguooooo/openapi-lark openapi-lark sync --parallel 4
```

工具会自动算每个接口的 sha256，没变的接口直接 skip。**只改了 1 个接口的话几秒就回**：

```
| service     | status | doc                                | duration |
| my-service  | ✓ ok   | endpoint: 1 ok / 0 failed /        | 5.4s     |
|             |        | 0 warning / 166 skipped across 167 |          |
```

### 强制全量重推

```bash
... openapi-lark sync --force
```

跳过 hash 缓存。一般不需要；上线 v1.7+ 渲染规则变化后想刷新所有 doc 才用。

### 本地预览不推送

```bash
... openapi-lark render my-service
# 生成到 ./.openapi-lark/my-service/<tag>/<接口>.md
```

或：

```bash
... openapi-lark sync --dry-run
```

### 诊断环境

```bash
... openapi-lark doctor
```

会检查 lark-cli 装没装、版本对不对、auth 状态、各 docToken 的可读性。

## 常见问题

### Q1：父节点标题被改成「Authentication」/「Untitled」

**原因**：早期版本同步时 widdershins 的 `# Authentication` H1 把 docx title 偷了。

**解法**：在 config 加 `parentTitle: <你想要的名字>`，下次 sync 自动改回来。

### Q2：`Error: missing required scope(s): wiki:node:write`

**解法**：

```bash
lark-cli auth login --scope "wiki:node:write wiki:node:move"
```

`wiki:node:move` 也加上（清理 zombie 节点时要用）。

### Q3：响应字段表只看到 default 没看到 200

**原因**：你的 openapi 用 `allOf: [BaseResponse, {data:...}]` 包装响应。早期 widdershins 不展开 allOf。

**解法**：v1.7+ 已自动扁平化。如果还是看不到：

```bash
rm .openapi-lark/sync-lock.json
... openapi-lark sync
```

强制重渲一次。

### Q4：「server time out error」

**原因**：单个 docx 太大（>1MB），飞书 server 拒收。

**解法**：
- 确认 config 里 `mode: endpoint`（不是 single 或 tree）
- endpoint mode 下每接口独立 doc 一般 10-30KB，不会撞上限
- 如果是 tag 中间节点超限了：升级到最新 openapi-lark（v1.7+ 把 tag 中间改成短 TOC 了）

### Q5：飞书 wiki 出现一堆「Untitled」/「Authentication」副本节点

**原因**：早期 sync 中途失败/早期版本 bug 留下的孤儿节点。

**解法**：

```bash
# 通过 lark-cli wiki +move 把它们移到别的 wiki space（眼不见为净）
lark-cli auth login --scope "wiki:node:move"
# 然后手工 move，或在飞书 wiki UI 一键删
```

### Q6：能不能让某些接口不同步？

可以用 includeTags / excludeTags：

```yaml
services:
  - name: my-service
    ...
    excludeTags: [internal, deprecated]   # 这些 tag 的接口不推
```

### Q7：能在 CI 跑吗？

可以。CI 需要：
- `lark-cli` 装好
- 通过环境变量做 auth（`LARK_APP_ID` / `LARK_APP_SECRET`）—— 这部分 lark-cli 文档里有
- 你的 docToken 用 env 引用：

```yaml
services:
  - name: my-service
    docToken: ${LARK_DOC_MY_SERVICE}
```

CI 注入对应 env 即可。

## 设计原则

了解这些有助于排查 corner case：

1. **文档由工具拥有**：sync 会清空目标 docx 并重写。**不要在 docx 里手写内容**，会丢。讨论协作请用别的 wiki 节点。

2. **lark docx 没有真正的「折叠块」**：markdown / XML 都没有 collapse 容器。JSON 示例放在 `### 响应示例` 子标题下，用户通过 wiki 大纲折叠。

3. **每个 docx 唯一一个 H1**：lark 用 markdown body 第一个 `# H1` 当 title。本工具把所有其他 H1 降级为 H2，并强插我们要的 H1。

4. **allOf 扁平化是自动的**：渲染前预处理 schema，把 allOf 合并成单 object。后端 BaseResponse 模式无感支持。

5. **hash 缓存按 docToken**：sync-lock.json 存的是 `{docToken: sha256}`。改 config 名字或换 docToken 就不命中缓存了。

## 进阶

### 多服务一个仓库

```yaml
services:
  - name: user-svc
    openapi: services/user/openapi.yaml
    docToken: doccnAAAAAAAAA
    mode: endpoint
    parentTitle: 用户服务
  - name: order-svc
    openapi: services/order/openapi.yaml
    docToken: doccnBBBBBBBBB
    mode: endpoint
    parentTitle: 订单服务
```

一次 `sync` 同步所有。`--parallel 4` 并发跑。

### 跨仓库共用基础配置

```yaml
# .openapi-lark.yaml 在子项目里
extends: ../shared/openapi-lark-base.yaml
services:
  - name: my-svc
    ...
```

### 多环境

```yaml
services:
  - name: my-svc
    docToken: ${LARK_DOC_MY_SVC_PROD}    # 用 env 切换
    ...
```

`LARK_DOC_MY_SVC_PROD=doccnXXX npx ... sync`

## 卡住了？

1. 跑 `openapi-lark doctor` 看一眼基础设施
2. 用 `--dry-run` 只渲染不推，先验本地输出
3. 看 `./.openapi-lark/<svc>/` 下生成的 md，确认渲染对不对
4. 看本仓库 [SKILL.md](./SKILL.md) 的「故障诊断序」节
